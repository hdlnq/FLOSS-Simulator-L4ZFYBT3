# OSS コミュニティ生存シミュレーション — 実験レポート

## 1. 目的

エージェントベースシミュレーション (ABS) を用いて、OSS コミュニティの生存率に影響を与える要因を定量的に分析する。特に以下の 3 つの変数が生存率に与える影響を比較検討した。

- 初期 Committer 数
- 新規 Contributor の流入率
- 昇格制度の有無

---

## 2. シミュレーション概要

### モデルの概要

OSS コミュニティを Committer と Contributor の 2 種類のエージェントで構成する。各エージェントは Motivation `[0.0, 1.0]` を持ち、毎日減衰する。Motivation が低いほど Inactive になりやすく、全 Committer が Inactive になった時点でプロジェクトは「死亡」とみなす。

### 基本パラメータ (全シナリオ共通)

| パラメータ | 値 |
|---|---|
| `MaxSteps` | 1825 日 (5 年) |
| `SimulationRuns` | 10,000 |
| `InitialContributors` | 0 |
| `ActivityDecayRate` | 0.01 /日 |
| `DeactivationRate` | 0.10 |
| `ActivationRate` | 0.20 |
| `CommitProbability` | 0.40 |
| `CommitActivityBoost` | 0.15 |
| `ReviewProbability` | 0.60 |
| `MergeAcceptRate` | 0.70 |
| `PromotionThreshold` | 0.80 |
| `PromotionProbability` | 0.01 |
| 初期 Motivation | `[0.0, 1.0]` 一様乱数 |

---

## 3. 実験設計

ベースライン条件に対して 1 変数ずつ変更し、生存率と平均死亡日を比較した。

| シナリオ | `InitialCommitters` | `NewContributorLambda` | `EnablePromotion` |
|---|---|---|---|
| ベースライン | 1 | 0.01 | true |
| A: Committer 増加 | **3** | 0.01 | true |
| B: 流入増加 | 1 | **0.2** | true |
| C: 昇格なし | 1 | 0.01 | **false** |

---

## 4. 結果

| シナリオ | 生存率 | 平均死亡日 | 標準偏差 |
|---|---|---|---|
| ベースライン | 22.1% | 78.7 日 | ± 143.9 日 |
| A: Committer 増加 | **78.0%** | 132.5 日 | ± 174.2 日 |
| B: 流入増加 | 39.9% | 24.1 日 | ± 45.4 日 |
| C: 昇格なし | **0.5%** | 191.9 日 | ± 301.4 日 |

---

## 5. 考察

### 5.1 初期 Committer 数が生存率に最も強く影響する (ベースライン 22.1% → シナリオ A 78.0%)

初期 Committer を 1 人から 3 人に増やすだけで、生存率が 22.1% から 78.0% へと大幅に向上した。これは単純な人数の問題ではなく、**冗長性**の効果によるものと解釈できる。

- Committer が 1 人の場合、その 1 人が Inactive になった瞬間にプロジェクトが死亡する
- Committer が 3 人いれば、1 人が Inactive になっても残りの 2 人が活動を継続できる
- 3 人が同時に全員 Inactive になる確率は確率の積となり、急激に低下する

実際の OSS プロジェクトにおいても「1 人依存プロジェクト」は放棄リスクが高いとされており、このシミュレーションはその直感を定量的に裏付けている。

### 5.2 流入増加は生存率を上げるが、副作用がある (ベースライン 22.1% → シナリオ B 39.9%)

新規 Contributor の流入率を 0.01 から 0.2 に引き上げると生存率は 39.9% に改善した。しかし同時に、**平均死亡日が 78.7 日から 24.1 日へと大幅に短縮**するという逆説的な結果が現れた。

この原因として以下が考えられる。

- 新規 Contributor が増えると PR 投稿が増加する
- Committer 1 人では急増した PR をさばききれず、放置 PR が大量発生する
- 放置 PR のペナルティ (`PrIgnoredActivityPenalty`) により Contributor の Motivation が低下
- さらに、Committer 自身もレビュー負荷が増すことで消耗が加速する可能性がある

生存した約 40% の run は、昇格が早期に成功して Committer 数が増えた run だと考えられる。流入増加は「昇格候補の母数を増やす」という意味では有効だが、**Committer の処理能力を超えると逆効果になる**ことを示している。

### 5.3 昇格制度がなければ生存率はほぼ 0% (シナリオ C)

昇格を無効にすると、生存率は 0.5%（10,000 run 中 50 run のみ生存）にとどまった。これは Committer 1 人スタートという条件下では**昇格が生存のほぼ唯一の経路**であることを意味する。

興味深いのは、昇格なしのシナリオでの平均死亡日 191.9 日がベースライン (78.7 日) より長い点である。これは以下のように説明できる。

- 昇格ありの場合、Contributor の参加によって PR キューが生まれ、放置ペナルティが Motivation を削る
- 昇格なしの場合は Contributor の行動が Committer に影響を与えないため、Committer の Motivation が純粋に自然減衰のみで低下する
- 結果として「静かにゆっくり死ぬ」パターンになる

---

## 6. 結論

| 知見 | 実際の OSS への示唆 |
|---|---|
| 初期 Committer 数が生存率に最も強く影響する | プロジェクト立ち上げ時に複数のコアメンバーを確保することが重要 |
| 流入増加は昇格が追いつかないと逆効果になる | Committer のレビュー体制を整えてから外部への宣伝・募集を行うべき |
| 昇格制度がなければ長期生存は不可能 | コントリビューターをコアメンバーに育てる仕組みが不可欠 |

---

## 7. 今後の課題

- Committer 増加 × 流入増加を組み合わせた場合の相乗効果の検証
